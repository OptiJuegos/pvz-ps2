Refactorización completa del backend gráfico de Plants vs. Zombies para PlayStation 2
Objetivo

Refactorizar completamente el backend gráfico de la versión de PlayStation 2 de Plants vs. Zombies con dos objetivos principales:

Reducir significativamente el consumo de memoria (RAM y VRAM).
Mejorar la arquitectura del código para facilitar futuras optimizaciones específicas de PS2.

Actualmente el backend funciona correctamente y ya incorpora numerosas optimizaciones, pero gran parte de ellas fueron agregadas de forma incremental durante el desarrollo del port. Como consecuencia, gran parte de la lógica terminó concentrándose dentro de gles_ps2.cpp, que hoy contiene responsabilidades muy diferentes mezcladas entre sí.

El objetivo no es únicamente optimizar el rendimiento, sino transformar el backend en un diseño modular, mantenible y preparado para seguir evolucionando.

Estado actual del proyecto

Durante el desarrollo del port ya se implementaron numerosas optimizaciones específicas para PlayStation 2.

Entre ellas:

Lazy loading de imágenes.
Decodificación bajo demanda desde los archivos PAK.
Imágenes "stub" que sólo almacenan dimensiones hasta su primer uso.
Sistema de purgado de imágenes entre cambios de pantalla.
Recarga automática cuando una imagen vuelve a utilizarse.
Arena lineal de VRAM.
Reutilización de bloques de VRAM.
Upload diferido de texturas.
Eliminación del Z-buffer para recuperar VRAM.
Corrección de múltiples bugs relacionados con gsKit.
Corrección del bug del ALLOC_ERROR.
Corrección del cast inválido entre ResourceManager y TodResourceManager.
Downscale configurable para reducir presión sobre VRAM.
Compatibilidad con renderizado ortográfico de PvZ.
Cursor software.
Varias correcciones del backend OpenGL para PS2.

Todas estas optimizaciones funcionan, pero muchas terminaron mezcladas en el mismo archivo.

Problema arquitectónico

Actualmente gles_ps2.cpp contiene lógica correspondiente a prácticamente todo el renderer:

gestión de texturas
subida a VRAM
administración del heap de VRAM
creación de texturas
renderizado
batching
shaders fijos
conversión de formatos
paletas
uploads parciales
allocator
estados OpenGL
helpers de conversión
backend GS
sincronización
etc.

El archivo se volvió excesivamente grande y complejo.

Cada nueva optimización implica modificar código perteneciente a múltiples subsistemas diferentes.

Esto aumenta muchísimo el riesgo de introducir regresiones.

El objetivo es separar claramente responsabilidades.

Refactorización propuesta

La intención es dividir el backend en módulos independientes.

Por ejemplo:

administrador de VRAM
administrador de texturas
upload de texturas
conversión de formatos
manejo de paletas
backend GS
backend OpenGL
render state
batching
render targets
utilidades específicas de PS2

No es obligatorio seguir exactamente esta división, pero la idea general es eliminar el enorme archivo monolítico actual.

Cada módulo debería tener una responsabilidad bien definida.

Refactorización del sistema de imágenes

Además de reorganizar el renderer, se desea rediseñar completamente el almacenamiento interno de imágenes.

Actualmente todas las imágenes terminan convertidas inmediatamente a RGBA8888.

Es decir:

PNG
↓

RGBA8888

↓

Image::mBits

Lo mismo ocurre para JPEG y otros formatos.

Esto implica que toda imagen consume inmediatamente 4 bytes por píxel.

En una PC esto apenas importa.

En una PlayStation 2 representa varios megabytes desperdiciados.

Problema observado

Ejemplo real:

HouseofTerror20Outline.png

3177 x 43

PNG PALETTE 8 bits

Actualmente termina ocupando aproximadamente

0.52 MB

únicamente porque el loader la expande completamente a RGBA8888.

Sin embargo originalmente era una imagen indexada.

Su representación compacta ocupa una fracción de esa memoria.

Este patrón ocurre con numerosos recursos del juego.

Nueva filosofía

La representación interna de una imagen no debe depender del renderer.

Debe depender del formato más eficiente para almacenarla.

El renderer debe ser capaz de consumir distintos formatos internos.

El almacenamiento y el renderizado dejan de estar acoplados.

Objetivos del nuevo sistema

Cada imagen debe permanecer en el formato más compacto posible.

Por ejemplo:

PNG indexados deben permanecer indexados.
PNG RGB deberían mantenerse sin expandir innecesariamente.
JPEG deberían almacenarse en una representación compacta apropiada.
Sólo imágenes que realmente necesiten alpha completo deberían utilizar almacenamiento RGBA.

La expansión a RGBA8888 debe producirse únicamente cuando sea realmente necesaria.

No durante la carga.

Compatibilidad

Todo el motor existente debe seguir funcionando.

Las APIs públicas no deberían modificarse.

El resto del código debe continuar utilizando Image, MemoryImage y GLImage exactamente igual que ahora.

Los cambios deben permanecer encapsulados dentro del backend gráfico.

Compatibilidad con el lazy loading

La nueva arquitectura debe integrarse completamente con el sistema de lazy loading ya implementado.

Debe seguir siendo posible:

cargar únicamente dimensiones
decodificar bajo demanda
purgar memoria
volver a decodificar automáticamente
reconstruir texturas
mantener imágenes "pinned"
soportar imágenes modificadas dinámicamente

Toda esta lógica debe conservarse.

Compatibilidad con VRAM

El nuevo sistema debe seguir funcionando con:

arena lineal de VRAM
uploads diferidos
reciclado de bloques
flush automático al agotarse la VRAM
recreación de texturas después de un purge

No debe romper el administrador de VRAM existente.

Compatibilidad con gsKit

Debe mantenerse la compatibilidad con:

gsKit modificado
GSKIT_ALLOC_ERROR == 0
renderizado ortográfico
doble buffer
framebuffer CT16
texturas CT32
backend OpenGL actual
Objetivos de calidad del código

Se prioriza una arquitectura limpia por encima de implementar soluciones rápidas.

Se busca:

responsabilidades bien separadas
clases pequeñas
módulos reutilizables
código fácil de mantener
menor acoplamiento
eliminación de lógica duplicada
nombres claros
comentarios explicando las decisiones específicas de PS2
Objetivo final

No se busca únicamente ahorrar memoria.

Se busca convertir el backend gráfico de PS2 en una implementación sólida y modular, donde cada subsistema tenga una responsabilidad clara y donde futuras optimizaciones (como PSMT8, nuevos formatos de textura, atlas, compresión, streaming de recursos o mejoras del renderer) puedan incorporarse sin seguir aumentando la complejidad de gles_ps2.cpp.

El resultado esperado es un backend específicamente diseñado para las limitaciones de PlayStation 2, con una arquitectura mantenible, extensible y significativamente más eficiente tanto en RAM como en VRAM que la implementación actual.
